This One Weird Trick Makes Developing iOS Apps Against a Local Server Way Easier

I want to talk about something that‚Äôs always annoyed me about the iOS development process, and an elegant way to fix it.

OK. So let‚Äôs say you‚Äôre developing an oh-so-fancy app that talks to a server - say, api.myfancyapp.com. It can be helpful to run a copy of your API server locally on your Mac, and then point your app to that instead (especially if you‚Äôre developing features in tandem in your app and on the server). So now instead we want our app to point to, say, localhost:3000. To fix this, you can create a custom Debug build configuration, and make it so that your app‚Äôs API client pulls its base URL dynamically from that build configuration. This process is fairly well-covered elsewhere (I like this blog post - it‚Äôs old and written in pre-ARC objective-c, but explains everything nicely), so I won‚Äôt go into too much detail here.

Except, what if you want to test your app on a device? While the iOS simulator is smart enough to understand that localhost means ‚Äúthe Mac on which the simulator is running‚Äù, if your app is running on a device it‚Äôll try and connect to itself (and unless you‚Äôre way cleverer/insaner than I am, you probably don‚Äôt have a rails server running on your iPhone) and fail. One nice solution to this problem is to use a service like ngrok, which allows you to forward a port from your local machine onto the public-facing internet and then connect to that instead. You could also use the awesome Charles Proxy to redirect all traffic from your phone through your Mac. Both of these approaches have their drawbacks, though - you need to buy ngrok/Charles, need to remember to start them each time you start debugging, etc. Also, neither approach scales well when you have multiple collaborators on the same app, which I‚Äôll address more in a second.

Alternatively, another really clever thing you can do is to connect your Mac and your iOS device to the same WiFi network and then point your iOS device to your Mac‚Äôs ‚ÄúLocal Network Name‚Äù instead. You can see what this value is by opening the ‚ÄúSharing‚Äù pane in System Preferences, or by running scutil --get LocalHostName in a terminal and appending .local to the result. So, instead of setting your app‚Äôs API client‚Äôs base URL to localhost:3000, you‚Äôd set it to something like jacks-macbook.local (that‚Äôs what my value is, anyway), and it‚Äôll work in both the simulator and on a device!

Except, what if you have multiple people working on the same app? You can‚Äôt just hard-code jacks-macbook.local into your app‚Äôs source code - then it won‚Äôt work for your colleagues, and they‚Äôll change it, and you‚Äôll get merge conflicts, and eventually all of you will get frustrated and quit your jobs (ok, hopefully not that last part). Also, this base URL is really just configuration, and shouldn‚Äôt live in your source code. So instead, we can harness the power of custom Xcode schemes. If you haven‚Äôt worked with them that much, Xcode schemes are a really nice way of defining a sort of saved, custom ‚Äúlaunch environment‚Äù - you can specify environment variables and launch arguments to run your app with every time. Importantly, you can also specify custom scripts to run each time you build your app (for more information, there is (of course) a nice NSHipster article on this. My friend Cameron also goes into some good detail on his blog).

So, here‚Äôs the plan: we‚Äôll make a new scheme that marks our build as being in a local development environment. We‚Äôll then add a build script that, when in that environment, determines your Mac‚Äôs local network name and embed that value into your app so that it‚Äôs accessible at runtime. That way, if 2 different developers build the app with our new scheme, each copy of their app will automagically point itself to the appropriate URL.

So to do this, first let‚Äôs create a new scheme (Product ‚Üí Scheme ‚Üí Manage Schemes‚Ä¶ in Xcode, then click the ‚Äú+‚Äù in the lower left) and name it MyFancyApp-LocalDevelopment. Make sure the ‚Äúshared‚Äù box is checked (this will allow you to check this new scheme into source control, otherwise it‚Äôll just live locally on your machine). Then select it and hit ‚ÄúEdit‚Ä¶‚Äù. Click the arrow next to ‚ÄúBuild‚Äù on the left, then select ‚ÄúPre-actions‚Äù, then the ‚Äú+‚Äù in the lower left, then select ‚ÄúNew run script action‚Äù (side note: this workflow is really hard to discover). Then, paste the following script in (make sure you‚Äôve selected your app‚Äôs target in ‚Äúprovide build settings from‚Äù, or this won‚Äôt work):

touch "${CONFIGURATION_TEMP_DIR}/LocalDevelopmentOverridesEnabled"

Next, do the same thing, this time adding a script to ‚ÄúPost-actions‚Äù with the following contents:

rm "${CONFIGURATION_TEMP_DIR}/LocalDevelopmentOverridesEnabled"

These two scripts simply create a file in the build‚Äôs intermediates directory (a generally OK place to put things like this) before the build starts and delete it after the build is done. Thus, we can use the presence (or absence) of this file during our build to know if we‚Äôre running in the MyFancyApp-LocalDevelopment scheme.

Next, go to the ‚ÄúBuild Phases‚Äù tab of your application‚Äôs settings, click the ‚Äú+‚Äù in the upper left, and add a new ‚ÄúRun Script Build Phase‚Äù with the following contents (make sure it‚Äôs executed at the very end of the ‚ÄúBuild Phases‚Äù section):

PLIST_FILE="${TARGET_BUILD_DIR}/${CONTENTS_FOLDER_PATH}/LocalDevelopmentOverrides.plist"
rm -f $PLIST_FILE

if [ -f $"${CONFIGURATION_TEMP_DIR}/LocalDevelopmentOverridesEnabled" ]
then
    LOCAL_NETWORK_NAME="$(scutil --get LocalHostName | tr "[:upper:]" "[:lower:]").local"
    /usr/libexec/PlistBuddy -c "add :localNetworkName string $LOCAL_NETWORK_NAME" $PLIST_FILE
fi

This script checks for the presence of the ‚Äúare we running in the local development scheme‚Äù file, and if it‚Äôs there, it creates a new plist file named LocalDevelopmentOverrides and adds your Mac‚Äôs local network name to it under the localNetworkName field (incidentally, it does so with a nice but not-well-known tool called ‚ÄúPlistBuddy‚Äù, for which you can find a nice explanation here. This file is created in your target‚Äôs build directory (i.e. your DerivedData folder), not your project directory, just before your application is copied onto your device (or simulator). It effectively injects a bit of configuration into your app at the last possible second. The nice thing about this approach is that it effectively makes it impossible to leak any sensitive information into your application when you‚Äôre not running in this LocalDevelopment scheme - you don‚Äôt even have a file to check into source control!

Finally, we can look for this LocalDevelopmentOverrides.plist file in our application at runtime as follows:

public struct LocalDevelopmentOverrides {

    public static let configuration: [String: String]? = {
        guard let path = NSBundle.mainBundle().pathForResource("LocalDevelopmentOverrides", ofType: "plist") else {
            return nil
        }
        return NSDictionary(contentsOfFile: path) as? [String: String]
    }()

}

Then, in your application, you could do something like:

if let hostNameOverride = LocalDevelopmentOverrides.configuration?["localNetworkName"] {
    myApiClient.baseUrl = "http://\(hostnameOverride):3000"
} else {
    myApiClient.baseUrl = "https://api.myfancyapp.com"
}

Neat! Again, the nice thing about this setup is that you can kind of forget about it: all you have to do to enter ‚ÄúLocal Development‚Äù mode is build your app in the MyFancyApp-LocalDevelopment scheme, and it‚Äôll auto-infer your Mac‚Äôs local network name and embed it properly, etc.

Bonus round: Application Transport Security

While writing this up, I ran into a small, related issue that I felt like others were likely to encounter as well. iOS9 introduces a new feature called Application Transport Security, which effectively mandates that your app only communicates over appropriately-up-to-date HTTPS connections. It‚Äôs a good thing, and it‚Äôs on by default, and you should leave it that way! Except, if you followed along with this blog post so far, it will completely and utterly break when you‚Äôre running your app in Local Development mode, as we‚Äôre connecting to our Mac over HTTP, not HTTPS (your Mac most likely isn‚Äôt configured to do that to begin with). So, you‚Äôll get a warning like: ‚ÄúApp Transport Security has blocked a cleartext HTTP (http://) resource load since it is insecure. Temporary exceptions can be configured via your app‚Äôs Info.plist file.‚Äù

Hmm, temporary exceptions you say? As it turns out, you can tell your app to not worry about HTTPS for specific domains by specifying appropriate values in your applications‚Äôs Info.plist (you can also disable App Transport Security entirely, but you shouldn‚Äôt do this. You can find more information in this blog post). In fact, we can use all of our new Local Development tools to add the necessary information to our Info.plist just before the app is copied to your device. Thus, we‚Äôll only whitelist your development domains when, well, in development.

To do this, simply add another ‚ÄúRun Script Build Phase‚Äù in your application settings‚Äô Build Phases tab with the following contents (which lean heavily on our new friend PlistBuddy):

if [ -f $"${CONFIGURATION_TEMP_DIR}/LocalDevelopmentOverridesEnabled" ]
then

INFO_PLIST="${TARGET_BUILD_DIR}/${INFOPLIST_PATH}"
LOCAL_NETWORK_NAME="$(scutil --get LocalHostName | tr "[:upper:]" "[:lower:]").local"

/usr/libexec/PlistBuddy -c "Add :NSAppTransportSecurity:NSExceptionDomains:$LOCAL_NETWORK_NAME:NSIncludesSubdomains bool true" $INFO_PLIST
/usr/libexec/PlistBuddy -c "Add :NSAppTransportSecurity:NSExceptionDomains:$LOCAL_NETWORK_NAME:NSThirdPartyExceptionAllowsInsecureHTTPLoads bool true" $INFO_PLIST

fi

exit 0

Two small, annoying notes here - one, it looks like App Transport Security doesn‚Äôt handle uppercase characters well, so we‚Äôre using tr to lowercase your mac‚Äôs local network name (it‚Äôs case-insensitive on the OS X side of things anyway). Two, Xcode appears to aggressively cache the generated Info.plist in your target‚Äôs build directory, meaning that these injected values might stick around longer than they should, even if you change schemes. To fix this, just add a final Run Script Build Phase, this one just before the ‚ÄúCopy Bundle Resources‚Äù phase, containing the line:

rm -f "${TARGET_BUILD_DIR}/${INFOPLIST_PATH}"

This simply deletes the cached version of that file, and forces Xcode to re-copy it during the next build.
